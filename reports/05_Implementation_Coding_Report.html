1. Introduction

This report documents the complete implementation phase of FitPulse with emphasis on production-grade authentication, cloud-backed user data, and deployment stability. The coding work evolved from a UI-first prototype to a secured and deployable web application with Firebase integration.

1.1 Implementation goals

  • Implement reliable user authentication with email/password and Google sign-in.
  • Persist user profile and onboarding data in Firestore with secure access rules.
  • Improve runtime resilience by handling popup, redirect, and profile-sync edge cases.
  • Move environment-sensitive configuration outside source code and into environment variables.
  • Deliver a production deployment that is verifiable through live and repository links.

3. Architecture and Module Design

3.1 Frontend composition

The application is organized into pages, reusable components, data helpers, and a centralized AppContext. Screen transitions and data updates are driven by reducer actions to avoid inconsistent state mutations.

4. Core Implementation Details

4.1 Authentication workflows

4.2 State management and routing

The reducer in AppContext handles NAVIGATE, LOGIN, RESTORE_SESSION, LOGOUT, COMPLETE_ONBOARDING, START_WORKOUT, and END_WORKOUT actions. Route restoration is controlled by auth readiness and onboarding status, preventing landing-page loops after successful sign-in.

4.3 User profile and onboarding persistence

  • ensureUserProfile creates or merges Firestore profile data on auth state changes.
  • createUserViewModel merges profile values with safe defaults for UI rendering.
  • completeOnboarding writes onboarding selections and stats to Firestore.
  • Profile page now displays authenticated user data instead of static mock values.
